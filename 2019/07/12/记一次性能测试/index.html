Jmeter 的组成：

Jmeter 的组成是由一个大的测试计划，下面是一个线程组(即测试场景),
线程组又包含Sampler，断言，定时器，监听器，配置元件，后置处理器，前置处理器，逻辑控制器
的这样一个结构。

所以整个的配置参数都是基于线程组创建的。

性能测试中主要关注的点

我先概况下，主要关注的地方为：

  • 聚合报告

  • jp@gc - PerfMon Metrics Collector （服务器信息）

  • 程序日志

  • SQLServer数据库

  • IIS 中该应用的工作进程

  • 网络连接数

聚合报告

首先看的就是聚合报告，在启动测试计划时就要通过聚合报告看线程有没有堵塞卡顿的情况（所有线程是不是都在跑）

聚合报告中主要关注平均响应时间、最大响应时间（单位：毫秒）以及
Error，如果出现 Error
也就没多大意义了，可停止测试，通过开启察看结果树和日志排查是服务器问题还是程序问题。我所遇到的服务器问题有：

  • 并发量太多，端口不够。设置 TCP/IP 端口可用值

  • IIS 服务器队列长度不够。设置 IIS 中服务器队列长度

服务器信息

CPU、内存在整个测试过程中的负荷，有没有持续飙高

网络有没有长时间等待处理的网络包

程序日志

请求返回是否都正常。

也可以通过日志查看线程有没有堵塞卡顿（请求和返回是一条一条过来的，还是一大段请求一大段返回）

SQLServer 数据库

对于平均响应时间、最大响应时间过长的，通过数据库监控，排查是否为 sql 性能问题

  1. 通过 SQL Server Profiler 工具抓取相关 sql，查看耗时时长

选择事件

设置筛选器，点击确定。当然，其他的参数也可以进行筛选

这里主要看 Duration 字段值

这里说下这几个字段的含义，从左到右分别是：

  • 事件分类，sql 语句，应用程序名称，操作系统用户，数据库用户，cpu
  • 占用率，读数据库次数，写数据库次数，执行脚本用时，应用程序进程号，开始时间，结束时间
  1. 通过活动监视器查看 右击服务器，选择 活动和监视器

主要看平均持续时间

找到执行时间较长的 sql，可使用执行计划分析该 sql 具体耗时的地方

IIS 工作进程

通过 IIS 工作进程，排查是否有 IIS 导致的性能问题

打开IIS ，双击 工作进程

双击该进程

查看经过的时间

网络连接数

连接数的增多意味着对系统内存资源的消耗越多，一方面也体现了防火墙一定的能力

可通过 netstat -an | find "端口"

也可通过性能监视器中，计数器中的 Current Connections 查看

注意点

  • 测试准备时，对数据量的大小也要进行一个统计。数据量对性能有很大的影响

  • 正式测试时，将察看结果树禁用，避免性能损耗

  • 在测试时，避免进行其他操作

  • 一轮并发量跑完之后，记得清空监听器中的记录，以准备下一轮的并发量测试

彩蛋

tempdb 即临时数据库，增加适当数据文件，可用于分担数据库执行、查询等的负担

IIS 遇到资源加载瓶颈，可在应用池增加该应用的工作进程数，但可能存在一定的负面效果。一个比较好的设想是使用两台服务器的方式进行分担。
